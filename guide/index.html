How this was built

Everything on the main page — copy, art direction, 8 generated illustrations, 4 animated loops, the custom Three.js scene, and the deploy — was produced by Claude (Anthropic) inside Claude Code, working from a single instruction, autonomously, end to end. This page is the honest methodology so you can reproduce it.

1. The brief

The starting point was one message: rebuild an existing client site (Capptions' "Seveso Control") as a standalone capability-demo — keep the brand and story, push the execution into 3D, generative imagery, and cinematic pacing, then ship it to GitHub Pages with this guide alongside it. No further back-and-forth until done.

2. Creative direction

Rather than jumping straight to code, a dedicated creative-direction pass ran first: a model was given the brand DNA (palette, type, the existing scroll-film metaphor) and the regulatory content (Seveso-III / BRZO's 7 VBS elements), and asked to return a structured brief — exact hex codes, named fonts, a concrete buildable 3D concept, a tightened narrative arc, and a full set of image-generation prompts written to read as one cohesive series.

Note: the brief was requested from Fable 5 first (this project's namesake), but that session was out of usage credits at build time, so Opus produced the brief instead. Worth knowing if you're trying to reproduce the exact creative voice.

3. Generating the imagery

Eight illustrations were generated from a single "style spine" paragraph appended to every prompt (palette, lighting, mood, format) so the set reads as one series instead of eight random images. Generation used GPT Image 2 via the claude-gpt-image-bridge skill, which drives OpenAI's image model through the codex CLI on an existing ChatGPT subscription — no separate API key.

~/.claude/skills/gpt-image-bridge/bin/gpt-image-2 "<prompt + style spine>" out.png --size 1536x1024

On this machine, codex's sandboxed shell couldn't perform its own final copy step (Windows policy), so a thin wrapper snapshotted ~/.codex/generated_images/ before and after each call and copied whatever new file appeared. If you hit the same "no PNG produced" error, that's the fix.

4. Animating key stills

Four of the eight stills became short (6s) seamless motion loops via the Higgsfield MCP, model kling3_0 in pro mode, silent (sound: off), each driven by a start-image + a motion-only prompt (no new subjects, just "what moves and how"). Flow: media_upload (presigned URL) → upload bytes → media_confirm → generate_video → poll job_status → download the result.

Only 4 of 8 stills were animated, deliberately — motion was spent on the beats where it deepens the story (the overload breathing, the drift gap widening, the structure self-assembling, the ambient connective texture), and left as static art everywhere else. More loops isn't better; matched loops are.

5. The 3D centerpiece

"The Drift Field" is hand-written Three.js — no template. ~5,000 THREE.Points plus two THREE.Line ribbons. Two driver values (drift, order) are set from GSAP ScrollTrigger progress; every frame, each particle's position is a lerp between a wandering position and one of 7 fixed lattice targets (the 7 VBS elements), so scrolling visibly organizes chaos into a ring. No shader code required for this look — a CPU-side mix() per particle is enough at this particle count, which keeps the whole effect debuggable in one file instead of a GLSL pipeline.

6. Assembly + iteration

The page itself is plain HTML/CSS/JS — Lenis for smooth scroll, GSAP ScrollTrigger for reveals and the 3D uniforms, no build step, so it deploys to GitHub Pages as-is. After the first full pass, it went through three independent fine-tooth review passes (fresh model context each time, no memory of having written the code) hunting specifically for: contrast/legibility failures, motion that doesn't earn its cost, pacing problems for a narrated video walkthrough, and brand-consistency drift from the locked palette/type. Findings were fixed between each pass.

7. Shipping it

gh repo create <you>/<name> --public
git init && git remote add origin <repo url>
git add -A && git commit -m "showcase" && git push -u origin main
gh api repos/<you>/<name>/pages -X POST -f "source[branch]=main" -f "source[path]=/"

GitHub Pages serves this as a static site with zero build pipeline — the /guide/ folder you're reading right now is just a second index.html at a sub-path, which is all a "route" is on Pages.

Reproduce it yourself

  1. Pick a real product/story you know well — generic placeholder content makes weak demos.
  2. Get a creative-direction brief written first, separately from the build — hex codes and named fonts up front save every downstream decision.
  3. Generate a small, cohesive image set (a shared style paragraph, not one-off prompts) before reaching for video.
  4. Animate only the moments that need motion. Static art is often the better choice.
  5. Build 3D as a value-mix between two known states driven by scroll — not a physics simulation — if you want it shippable in one sitting.
  6. Review your own work adversarially, more than once, before calling it done.
